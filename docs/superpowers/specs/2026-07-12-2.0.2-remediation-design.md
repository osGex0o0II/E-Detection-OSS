# E-Detection 2.0.2 Remediation Design

## Goal

Fix every issue confirmed by the 2026-07-12 multi-round review, add regression coverage for the affected boundaries, align the product on version `2.0.2`, and publish a verifiable unsigned `v2.0.2` GitHub Release.

## Release Contract

- `v2.0.2` is an intentionally unsigned release because no Authenticode certificate is available.
- The workflow, README, package metadata, installer instructions, updater text, and GitHub Release notes must all say that the artifacts are unsigned.
- SHA-256 checksums prove downloaded-byte integrity only. They must not be described as publisher identity or code-signing verification.
- The release must contain `EDetection-Setup-win-x64.exe`, `EDetection-win-x64.zip`, and `EDetection-win-x64.sha256.txt`.
- The tag, project version, assembly version, file version, release metadata, and displayed application version must resolve consistently to `2.0.2` or `2.0.2.0` as appropriate.

## Architecture

Use targeted extraction instead of rewriting the desktop application. Small pure helpers will own semantics that need direct regression tests, while the existing services retain orchestration responsibilities.

- `NativeDetectionBackendService` keeps the detection pipeline but gains safe enumeration, finite-value accounting, cancellation checkpoints, threshold invariants, chronological report aggregation, and atomic report publication.
- The desktop run coordinator starts the native backend away from the UI dispatcher while retaining UI-bound progress delivery.
- Version comparison moves to a semantic-version helper that handles prerelease precedence without discarding suffixes.
- Path ownership checks use canonical paths plus a trailing directory separator or exact equality; raw prefix and substring checks are prohibited for destructive operations.
- Poetry status uses the shared network/proxy path and remains opt-in during migration.
- Tests remain headless and runnable in CI. The current console smoke project becomes a named regression harness so each behavior can be run independently and observed failing before its fix.

## Detection Data Flow

1. Validate canonical input, output, and configuration paths before emitting `run_started`.
2. Enumerate CSV files with inaccessible directories ignored and reparse-point traversal disabled.
3. Read and parse each file with cancellation checks during input and row analysis.
4. Treat blank, unparsable, `NaN`, and infinity key measurements as invalid observations. A file with no finite key measurements cannot be reported as normal.
5. Validate configuration values and cross-field invariants before detection. `V_MIN_THRESHOLD` must not exceed `V_MAX_THRESHOLD`; `T_MIN_THRESHOLD` must not exceed `T_MAX_THRESHOLD`.
6. Aggregate details by device and compute first/last timestamps chronologically, independent of severity display order.
7. Build the workbook at a temporary path in the destination directory, close and validate the archive, then atomically move it to the unique final path. Delete the temporary file on cancellation or failure.

## Responsiveness And Cancellation

The WinUI command invokes the backend on a worker thread. `Progress<T>` continues to marshal events to the UI dispatcher. Cancellation is checked during file enumeration, CSV reading, row analysis, sorting, and report generation so a single large file cannot make the cancel command ineffective for the full file duration.

The implementation must not update observable UI collections from the worker thread. Shutdown cancellation remains idempotent, and no completion notification is emitted after cancellation.

## Local And Release Safety

- `Build-DesktopInstaller.ps1` accepts only output paths equal to or contained by the canonical runtime artifact root. A sibling such as `win-x64-backup` must be rejected before any deletion.
- Startup registry and scheduled-task cleanup must parse and compare the registered executable path exactly; `C:\AppBackup` must not match an installation at `C:\App`.
- Recursive source discovery must not follow directory junctions outside the selected input tree.
- Report write failures must not leave a final-name partial `.xlsx` file.

## Network And Privacy

Settings migration must never turn on poetry status for an existing user. When the user enables it, requests use the configured proxy policy and the same HTTPS-or-local endpoint validation used by other optional network services. Startup update checks remain controlled by their existing explicit setting.

## Regression Coverage

The headless test harness will cover at least these named cases:

1. blank and non-finite key measurements are not normal;
2. inverted voltage and temperature thresholds are rejected;
3. device first/last times are chronological after severity sorting;
4. cancellation interrupts a large single-file run;
5. inaccessible directories do not abort accessible CSV processing;
6. final report publication is atomic and failure cleanup removes temporary files;
7. prerelease-to-stable version precedence is correct;
8. installer output sibling-prefix paths are rejected without mutation;
9. startup cleanup does not match sibling-prefix executables;
10. settings migration keeps poetry disabled;
11. unsigned release wording is consistent across workflow and delivered documentation.

Each production change follows red-green-refactor: add one focused failing test, record the expected failure, implement the smallest behavior change, and rerun the focused and full suites.

## Formatting And Test Gates

- `dotnet format --verify-no-changes --no-restore` must pass. Existing line-ending drift will be normalized once, with generated files excluded.
- Debug and Release builds must complete with zero warnings and errors.
- Native backend, report, diagnostics redaction, release posture, package health, install, installer static, and available UI smoke tests must pass.
- `dotnet list ... package --vulnerable --include-transitive` must report no known vulnerable packages.
- The release artifact must pass package health after download, not only before upload.

## Publication Sequence

1. Merge the verified remediation branch into `main`.
2. Push `main` to `origin`.
3. Create and push annotated tag `v2.0.2` at the verified commit.
4. Let `.github/workflows/desktop.yml` build and publish the intentionally unsigned release.
5. Verify the GitHub workflow conclusion, release URL, tag commit, asset names, asset sizes, SHA-256 manifest contents, package metadata, and unsigned Authenticode state.

Publication is incomplete until those remote checks succeed. A local build, local tag, or uploaded workflow artifact alone is not a release.
